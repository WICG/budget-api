Abstract

This specification describes an API that can be used to retrieve the amount of budget an origin has available for resource intensive operations, as well as the cost associated with doing such an operation.

1. Introduction

This section is non-normative.

Web Applications have conventionally been able to execute code, make network requests and interact with the user by means of established interaction, usually through a browser tab. This has allowed users to associate the presence of a browser tab with the Web Applicationâs ability ability to do work on their behalf.

Following the introduction of the Push API [PUSH-API] and Web Background Synchronization [WEB-BACKGROUND-SYNC], this assumption no longer stands. Web Applications are now able to both trigger and schedule execution of code in the background, outside of the userâs control.

In an effort to mitigate risk to the user, user agents have implemented restrictions such as time limits on executing code in the background, or a requirement for the Web Application to use the Web Notification API [NOTIFICATIONS] to inform the user of the work theyâve done. Those restrictions are often unspecified and left up to the discretion of the user agent. In some cases, user agents will choose to not enforce these restrictions depending on the intensity of the userâs engagement with the Web Application.

This specification describes an API that exposes a budget that can be used by authors to determine their available budget for resource intensive background operations, as well as the cost associated with doing a certain action in the background.

Because this API relates to the ability to do work in the background, which is considered a privilege, functionality provided by this API is only available in a secure context.

This specification does not define how user agents establish or store the amount of available budget. It aims to define an API that exposes sufficient information to make the budget useful for authors, while not restricting the implementation details and heuristics specific to a user agent.

There are various use-cases for needing to know the immediately available budget:

The API defines a getBudget() method that can be used with getCost() to satisfy these use-cases, in unison with other APIs they depend on.

Determine whether a user visible interaction is required in response to a push message: 

self.addEventListener('push', event => {
    // Execute the application-specific logic depending on the contents of the
    // received push message, for example by caching the latest content.

    event.waitUntil(
      Promise.all([ navigator.getCost('push'), navigator.getBudget() ])
        .then(([cost, budget]) => {
            if (budget >= cost)
                return;  // No need to show a notification.

            // Not enough budget is available, must show a notification.
            return registration.showNotification(...);
        })
    );
});

2. Concepts

The user engagement with an origin is defined by the intensity of their interaction with the application by means of navigation, interaction and retention signals.

A background operation is the ability for an origin to execute potentially resource intensive code in the background.

The background operation cost is a positive number that describes the cost of executing a background operation on the userâs device.

An origin has an associated budget, which is a positive number derived from the user engagement that describes how many background operations the origin is able to do, depending on the associated background operation costs.

An origin may have an associated list of budget expectations. These are the expected budget values at given moments in time in the future.

3. Privacy Considerations

3.1. Applicability

Define that the intention is for this to be used for resource-sensitive operations (e.g. the Push API), not privacy-sensitive operations (e.g. Geolocation).

3.2. Permissions

Define that user agents may (1) offer the user means to override the budget available to a Web Application, and (2) the background operations themselves may be guarded behind additional permissions.

4. API

4.1. The NavigatorBudget interface

Should these methods be exposed as part of a navigator.budget object? 

[NoInterfaceObject,
 Exposed=(Window,Worker)]
interface NavigatorBudget {
  Promise<double> getCost(ActionType type);
  Promise<double> getBudget();
  Promise<sequence<BudgetState>> getBudgetDetails();
};

Navigator implements NavigatorBudget;
WorkerNavigator implements NavigatorBudget;

The getCost() method returns a promise that will be resolved with the background operation cost of the indicated background operation.

A user agent could lower the cost of certain background operations if the user has established a trust relationship with the Web Application.

When invoked, it MUST return a new promise promise and run the following steps in parallel:

  1. Resolve the promise with the background operation cost associated with the given type.

The getBudget() method returns a promise that will be resolved with the immediately available budget.

When invoked, it MUST return a new promise promise and run the following steps in parallel:

  1. Let origin be the entry settings objectâs origin.
  2. If the scheme of the origin is not https, reject promise with an SecurityError and terminate these steps.
  3. Resolve the promise with originâs budget when set, or zero otherwise.

The getBudgetDetails() method returns a promise that will be resolved with a sequence of BudgetState objects indicating the expected state of the budget at a given time in the future.

When invoked, it MUST return a new promise promise and run the following steps in parallel:

  1. Let origin be the entry settings objectâs origin.
  2. If the scheme of the origin is not https, reject promise with an SecurityError and terminate these steps.
  3. Let details be a new sequence.
  4. For each budget entry in originâs list of budget expectations:
    1. Let state be a new BudgetState instance.
    2. Set stateâs budget attribute to the budget value.
    3. Set stateâs expiration attribute to the DOMTimeStamp representing the final date of the validity of the budget.
    4. Add state to the details sequence.
  5. Resolve the promise with details.

4.2. The BudgetState interface

The BudgetState interface represents the known minimum amount of budget available at a specific point in the future. This enables authors to make near-term decisions about how to spend their budget. 

[Exposed=(Window,Worker)]
interface BudgetState {
  readonly attribute double budget;
  readonly attribute DOMTimeStamp expirationTime;
};

The budget attributeâs getter must return the budget at the associated expiration time.

The expiration attributeâs getter must return the timestamp representing the time, in milliseconds since 00:00:00 UTC on 1 January 1970, of the time the budget will expire.

4.3. The ActionType enum

The ActionType enumeration describes the known set of background operations that the Web Budget API caters for. Authors can use this in combination with getCost() to interpret their available budget as a quantifiable set of background operations it can be used for. 

enum ActionType {
  "push"
};

The following ActionType values are defined:

  • The push value represents a background operation in response to an incoming push message through the Push API. [PUSH-API]

Specifications are encouraged to extend the ActionType enumeration with their own values. Naming consistency with the Permission API [PERMISSIONS], when applicable, is recommended.
